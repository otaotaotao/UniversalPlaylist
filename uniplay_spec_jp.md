# UniPlay 仕様書 v0.2(名前は仮)

## 0. 動機・思想

**最小限の手間で、Apple Musicを使っている相手に、プレイリストを手渡す。** それだけをまず叶える。

ただし、単なるフォーマット変換ツールにはしたくない。念頭にあるのは:

- **日記のような音楽の受け渡し。** ライブラリに永続的に積み上がっていくコレクションではなく、流れてきて、聴かれて、流れていくもの。相手のApple Musicライブラリを圧迫しない。
- **絵葉書のような共有の感じ。** プラットフォームをまたぐ「同期」や「移行」ではなく、一通の手紙・一枚の葉書を送るような、軽くて個人的な行為として音楽を渡したい。
- だからこそ、便箋・蝋印・「P.S.」といったデザイン言語を選んでいる。これは装飾ではなく、上記の思想そのものの表現。
- **追記(気づき):曲ごとに一言コメントを添えたい、という欲求がある。** これは単なる機能追加ではなく、根っこにある動機の言語化かもしれない——プレイリストを渡すという行為の奥に、**「インティメイトでパーソナルなラジオ・ブロードキャスターごっこ」をしたいだけ**、という側面があるのではないか。1曲ずつに、なぜこれを選んだか・どう聴いてほしいか、DJが選曲の合間にひとこと挟むような声を残せる形にできると、この企画の本質により近づく気がする。

この思想は、今後の機能追加(自動プレイリスト生成、他プラットフォーム対応など)を評価する際の基準にもなる。「便利だが、ライブラリに恒久的に居座らせる」方向の機能は、都度この思想と照らして要検討。

## 1. 現在地:今日までに検証済みの現実

### 1.1 Spotify側

- **2026年2月11日、Spotify Web APIの仕様が変更された。** `GET /playlists/{id}/tracks` は `GET /playlists/{id}/items` に置き換わり、**曲の中身(items)は、ログイン中のユーザー本人が所有・共同編集しているプレイリストでしか返さない**という制限が付いた。Client Credentials Flow(ユーザーログイン不要な方式)では、もう曲の中身は一切取得できない。
- Development Modeのアプリは**認証ユーザー数の上限が5人まで**(2026年2月7日付の変更)。それ以上はExtended Quota Mode(法人登録・月間アクティブユーザー25万人以上などが条件)への申請が必要で、個人の趣味プロジェクトでは事実上届かない。
- アプリ作成者本人がSpotify Premium会員である必要がある(Development Modeの前提条件)。

→ 上記により、**「Spotifyリンクを貼るだけで自動取得」は、自前でOAuthログインを実装しても「5人まで」という上限からは逃れられない。**

### 1.2 Apple Music側

- Apple Music API(MusicKit)は、**検索を含む全操作に、有料Apple Developer Program($99/年)発行の署名付きトークンが必須。** 書き込みだけでなく検索だけでも要る。
- 一方、**Spotifyのような「5人まで」といった人数上限は見当たらない。** Web上でMusicKit JSを使う形であれば、$99を払えば何人でも各自のApple IDでログイン→各自のライブラリに書き込み、が可能と考えられる。
- **「次に再生(Play Next)」キューの操作は、$99を払っても実現できない。** これはApple Music純正アプリの再生キューであり、外部Webサイトが触れる範囲の外にある。MusicKitが制御できるのは「MusicKit自身の再生プレイヤー」であって、相手のiPhoneの純正Musicアプリの再生キューではない。

### 1.3 その他の技術メモ

- iTunes Search API(`itunes.apple.com/search`)は認証不要・無料で、曲のトップヒットへの直リンク(`trackViewUrl`)が取れる。ただしCORSで素直な`fetch`が通らない可能性があり、JSONPとの二段構えで実装している(2026年7月時点、実機での動作確認は未完了)。
- 既存の類似サービスの棲み分け:
  - **Soundiiz / TuneMyMusic / FreeYourMusic / SongShift**:プレイリスト丸ごとの本格変換サービス。FreeYourMusicはISRC優先マッチングという、今回目指す設計思想に近い実装をしている。
  - **Odesli(song.link)**:1曲・1アルバム単位のみ対応。プレイリスト単位では使えない。
  - **playlister.cc / Spotlistr**:プレーンなトラックリストをSpotify/Apple Music双方のプレイリストに変換。実際に試したところ精度は十分だったが、**曲順が前後する不具合**が観測された(並列検索後に元の順序へソートし直す処理が抜けている可能性を、反面教師として記録しておく)。
- Apple公式の「Transfer Music」機能(Apple Musicアプリ内蔵)は、相手が Spotifyで一旦保存→Apple Music設定から取り込み、という形で無料かつ今すぐ使える。ただし「一言リンクを送るだけ」という体験にはならない。
- **関連プロジェクト:Universal Playlist Format (UPF)**(`universalplaylist.stavros.io`)。2017年に提案された、プレイリストファイルの識別子(ファイルハッシュ・URI・MusicBrainz IDなど)を優先順位付きで持たせるフォーマット仕様。「複数の識別子を優先順位リストとして解決する」という発想は今回のISRC/MBID設計と同じ方向だが、実装は`pls2upl`という変換ツール1つのみでPre-alphaのまま停止しており、実質的にエコシステムは広がらなかった。同じ発想に先行者がいた、という記録として残しておく。

## 2. 現在のアーキテクチャ(モックアップ)

```
① Spotifyプレイリストを、Exportify等でCSVに書き出す(または、画面をスクロールしながら
   スクリーンショットを撮り、Claudeアーティファクト内でOCR的に抽出する版も試作済み)
        ↓
② CSVをHTMLファイルに直接埋め込む(データベース・バックエンド不要)
        ↓
③ 各曲について、iTunes Search API(曲名+アーティスト名+アルバム名で検索)から
   トップヒットの直リンクを取得。取得できなければ検索結果ページにフォールバック
        ↓
④ 静的HTML1枚として GitHub Pages にデプロイ
        ↓
⑤ 相手はページを開き、曲ごとの丸いアイコンボタンをタップ
   → 解決済みの曲は、ページ内(下からせり上がる埋め込みプレイヤー)でApple公式の
     embed.music.apple.com のiframeが開く。未解決の曲は検索ページを別タブで開く
```

### 2.1 UI仕様

- 便箋・ボトルメール風のデザイン(生成り色の紙、破線の仕切り、蝋印型のチェック)
- タイトル・メッセージ・曲ごとの一言コメントは、いずれもJS変数として分離済み(`playlistTitle` / `playlistMessage` / `notes`辞書)。配布前にこの変数だけ書き換えれば反映される。**「相手が編集できるテキストボックス」にはしていない**——静的サイトでバックエンドが無いため、相手側で編集しても保存されずリロードで消えてしまう。編集は常に配布前の作業として位置づけている。
- 曲ごとの重み付け(大・中・小の3段階)を導入。曲名をキーにした`weights`辞書で個別指定。小サイズはアルバム名を非表示にし、ボタン・チェックボックスも縮小(ただし箱のサイズ自体は変えず`transform: scale()`で見た目だけ縮小し、サイズが違っても中心線が揃うようにしている)
- 曲ごとに:曲名 / アーティスト名(改行) / 一言コメント(左に縦線、地の文から視覚的に分離)。アルバム名は埋め込みプレイヤー側に表示されるため、リスト上からは削除
- チェックボックスは行の左端、アイコンボタン(放送波のSVGアイコン)は右端。ボタンを押すと自動でチェックが入る(手動での取り消しも可能)
- 進捗の永続保存はしない(リロードで消える)。**これは技術的負債ではなく、「積み上がらない」という0章の思想とも相性が良い、意図的な選択として記録しておく。**

### 2.2 データ構造(現状)

```json
{
  "title": "string",
  "artist": "string",
  "album": "string",
  "url": "string | null",   // iTunes Search APIで解決済みの直リンク
  "resolved": "boolean",
  "done": "boolean",
  "note": "string | null",  // 曲ごとの一言コメント(実装済み)
  "size": "\"large\" | \"medium\" | \"small\""  // 重み付け(実装済み)
}
```

ISRC/MBIDは、CSV由来のデータには含まれていないため現状未実装(4章で再検討)。

## 3. 却下・保留したアーキテクチャ

| 案 | 状態 | 理由 |
|---|---|---|
| Spotify Web API直結(Client Credentials) | **却下** | 2026年2月の仕様変更で、曲の中身が原理的に取得不可に |
| Spotify OAuth(Authorization Code Flow) | **保留** | 技術的には動く(ログイン画面まで実装済み)。5人上限は個人利用なら許容範囲だが、仕様変更に追従し続けるメンテコストとの兼ね合いで保留 |
| Apple MusicKit JSでの自動プレイリスト生成・書き込み | **保留** | $99の投資判断が未確定。人数上限はない見込みで、投資対効果は悪くなさそうという整理まで完了 |
| 「次に再生」への自動追加 | **断念** | 外部Webアプリからは原理的に不可能と判明 |

## 5. 名前・語彙のメモ

- **現状の実装名:UniPlay**(フッター表記は "Universal Playlists")。ただしこれは仮で、正式には未確定。
- **名前は迷走中。** 今のところの最有力候補:
  - **Playlist Prayer**
  - **A Praylist for Us**
  - (検討過程で出た他の案:Praylist単体、Driftlist、Tideline、Sealed、Song Post、Dear Listener, など。方向性としては「pray/playの二重性」か「流れて積み上がらない(drift/tide)」のどちらかに寄っている)
- コードやファイル名は当面`uniplay`のままで進めて、名前が決まった時点で一括リネームする方針でよさそう。
- デザインモチーフ:便箋、message in a bottle、蝋印
- コピーの温度感:「P.S.」のような、手紙的な軽さ
